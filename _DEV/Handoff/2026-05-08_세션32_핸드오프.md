### C. 다부스 결제 409 충돌 근본 해결 (Seed 35, Supabase 직접 적용)

**증상**: 주류 + 타 매장 다부스 결제 시 두 번째 orders INSERT 가 `POST /rest/v1/orders 409 (Conflict)`.

**원인 분석**:
- `food_booths.booth_no` 중복 없음 (확인됨)
- 카운터 드리프트 없음 (확인됨, B03 last_no=7 = orders_max_no=7, drift=0)
- **진짜 원인**: pre-festival 기간 day_code=0 + 일별 카운터 리셋. festival_start=2026-05-15 이전 모든 날짜는 `GREATEST(0, (today - festival_start)::int + 1) = 0`. 그런데 `booth_order_counters` PK 가 `(booth_id, day_date)` 라 카운터는 매일 0 부터 리셋. → 어제 첫 주문 `M01-00001` + 오늘 첫 주문도 `M01-00001` → orders.order_number UNIQUE 충돌

**파일 백업**: `_DEV/Seeds/35_order_number_safety.sql` (gitignored, 적용 완료)

**해결**: `generate_booth_order_number` 를 collision-safe 로 강화. 함수 내부에서 후보 order_number 생성 → orders 테이블에 이미 존재하는지 NOT EXISTS 확인 → 있으면 카운터 한 번 더 증가 후 재시도 (1000 회 한도). day_code 가 같은 cross-day 충돌, booth_no 누락(prefix='X{md5 3자}'로 disambiguate), booth_no 중복(advisory lock 키를 booth_id 가 아닌 prefix 단위로) 모두 한 번에 방어.

**호환성**: order_number 포맷 그대로 (`{prefix}-{day_code}{NNNN}`, 끝 4자리 = counter). `parseOrderNumber` 호환 유지. booth_order_counters 스키마 변경 없음.

**검증**: 사용자 측 dev 머지 + 운영 적용 후 다부스 결제 재시도 필요 (다음 세션 항목).

## 6. 알려진 제약 / 운영 메모

### 6-1. 어드민 인증 한계
- Supabase auth 미사용. 하드코딩 계정 + sessionStorage. 보안보다 운영 단순성 우선.
- 새 도우미 계정은 `src/lib/adminAuth.ts` 코드 수정 + 배포 필요. 행사 중 동적 추가 불가.
- helper 는 `/help-desk` 만. super(`musanfesta`)는 모든 영역 + 결제 도우미도 가능.

### 6-2. 결제 도우미 정책
- 매장 정산은 method 무관 일률 `subtotal × 0.9626`. 매장 화면에 method 비공개.
- 식권 100% 결제 시 도우미가 method 입력해도 자동 `voucher_only` 분기.
- 외부 카드 결제 환불은 운영진이 단말기에서 별도 처리 (시스템은 정산 데이터에서만 제외).
- 현금 결제 환불은 시재 관리에 자동 반영 (cashOut 누적).

### 6-3. order_number 함수 강화 (seed 35)
- 행사 시작(5/15) 이후엔 day_code 가 1, 2, 3...으로 분리되어 cross-day 충돌 자연 소멸. 그래도 본 패치는 booth_no 중복 추가 / 카운터 드리프트 다른 케이스도 같이 방어하므로 유지.
- 1000 회 retry 한도 — 실 운영에서 도달하면 booth_order_counters 또는 orders 테이블 자체 손상 신호.

### 6-4. day_code = 0 (현재)
- 2026-05-08 = pre-festival. 모든 주문번호 day_code=0 (예: `B03-00007`).
- 5/15 부터 day_code=1. parseOrderNumber 끝 4자리 추출은 두 포맷 모두 호환.

### 6-5. 빌드 검증 주의
- 로컬 `tsc --noEmit` 은 incremental cache 사용 → 미사용 변수 / `noUnusedLocals` 위반 놓칠 수 있음.
- prod 빌드 (`tsc -b`)는 풀체크. push 전 캐시 클리어 후 재검증 권장:
  ```bash
  rm -f *.tsbuildinfo && npx tsc -b
  ```
