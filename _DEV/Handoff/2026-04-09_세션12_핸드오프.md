## 2. 핵심 결정사항

| 항목 | 결정 |
|---|---|
| **정산 기준** | 매출 = gross = `total_amount + discount_amount`. 쿠폰 할인은 플랫폼 부담. 매장 정산 시엔 부스 orders.subtotal 기준 (이미 gross) |
| **매출관리 탭 KPI** | 여기선 **손님 실지불 기준** (`payment.total_amount`) 유지. 정산 화면은 AdminOrders |
| **쿠폰 만료 모델** | `coupons.expires_at` 직접 필드. festivals.start/end_date 추가는 다음 세션 일차 탭 작업으로 미룸 |
| **쿠폰 race condition** | `markPaymentPaid` 에서 `UPDATE coupons SET status='used' WHERE id=X AND status='active'` 원자 전이. 이미 used 면 경고 로그만 (Toss 이미 승인) |
| **쿠폰 코드 포맷** | `MS-XXXXXX` (6자리 대문자 영숫자, 0/1/O/I 제외) = 32^6 ≈ 10억 공간 |
| **최소 주문 금액** | 쿠폰마다 개별 설정 가능 (`coupons.min_order_amount`). 기본 10,000원 |
| **쿠폰 관리 위치** | 운영 그룹 (대시보드·참가신청·쿠폰·공지). "매장 관리" 아님 — 프로모션 성격 |
| **시간 분석 범위** | 10~18시 고정 (9개 슬롯). 범위 밖 데이터는 테스트용으로 무시 |
| **시간차트 라벨** | SVG 밖 HTML div absolute positioning (viewBox 스케일 영향 없음) |
| **AdminOrders 리스트 금액** | gross 표시 + 쿠폰 칩 hover 툴팁으로 실지불/할인 노출 |
| **페이지네이션** | 15개/page. 최근 = 큰 순번. 공용 컴포넌트 |

## 7. 운영 테스트 시나리오 (커밋 전 확인용)

### 쿠폰 플로우
1. 어드민 → 쿠폰 관리 → 수동 발급 (금액 2000, 만료일 내일)
2. 쿠폰 코드 복사
3. 손님 → 메뉴 10,000원 이상 장바구니 → 체크아웃
4. 쿠폰 코드 입력 → 적용 → 할인 확인 (취소선+최종액)
5. 결제 → 성공
6. 쿠폰 관리 재접속 → 해당 쿠폰 사용완료 전이 확인
7. 주문/결제 관리 → 해당 payment 리스트 → gross 금액 + 쿠폰 칩 표시 확인
8. 상세 모달 → 원래/할인/최종 breakdown 확인

### 엣지 케이스
- 10,000원 미만 주문에 쿠폰 → "최소 주문 금액" 에러
- 한 번 쓴 쿠폰 재입력 → "이미 사용됨"
- 만료된 쿠폰 → "만료됨"
- 존재하지 않는 코드 → "존재하지 않음"
- 동시에 같은 쿠폰 두 결제 시도 → 한쪽만 성공, 다른 쪽은 경고 로그

### 환불 플로우
- 쿠폰 사용 건 환불 → payments+orders cancelled. 쿠폰은 used 유지 (refund 시 복구 로직은 아직 없음 — 다음 세션 검토)
