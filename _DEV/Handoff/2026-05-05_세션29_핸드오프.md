### 4-1. v3 결제 흐름 검증 (`meal_voucher_v3_payment.md` §7)

테스트 식권은 이미 발급된 상태 — 4번호 × 16장 (`01012345678` 8000원 11장 / `01023456789` 8000원 1장 / `01034567890` 10000원 1장 / `01045678901` 8000원 3장).

**케이스 A — 8,000원 식권 + 6,000원 주문 (잔액 소멸, PG 우회)**
- 결제 화면 라디오 식권 선택 → 잔액 2,000원 소멸 안내
- 결제 금액 0원, "식권 결제하기" 버튼
- 결제 후: `orders.voucher_consumed=6000`, `voucher_burned=2000`, `payments.payment_key=NULL`, `status='paid'`

**케이스 B — 8,000원 식권 + 10,000원 주문 (PG 2,000원)**
- Toss 결제창 2,000원
- `voucher_consumed=8000`, `voucher_burned=0`, `subtotal=10000`

**케이스 C — 할인쿠폰 + 15,000원 주문 (기존 동작)**
- PG 13,000원, voucher 컬럼 0

**케이스 D — 인솔교사 11장 → 5번 결제 후 "남은 장수 6장"**

**케이스 E — 다부스 비례 분배** (부스A 4,000 + 부스B 6,000 = 10,000 + 식권 8,000)
- 부스A.consumed ≈ 3,200 / 부스B.consumed = 4,800 (마지막 부스 잔여 흡수)
- SUM(consumed) = 8,000

**케이스 F — 다부스 + 잔액 소멸** (부스A 2,000 + 부스B 4,000 = 6,000 + 식권 8,000)
- 부스A.consumed = 2,000 / 부스B.consumed = 4,000 + burned = 2,000
- 매장 송금: subtotal × 0.9626

검증 SQL 은 핸드오프 §7-3 + 본 핸드오프 §6 참조.

```sql
-- 매장 송금 검증
SELECT SUM(subtotal) AS sales,
       SUM(subtotal) * 0.9626 AS booth_payout,
       SUM(subtotal) * 0.0374 AS toss_fee
FROM orders o JOIN payments p ON p.id = o.payment_id
WHERE p.status='paid' AND o.status NOT IN ('cancelled');

-- 정합성: 매장송금 = PG입금 + 운영자 순지출
SELECT
  SUM(o.subtotal) * 0.9626 AS lhs_booth_payout,
  SUM(p.total_amount) * 0.9626
    + (SUM(o.subtotal) - SUM(p.total_amount)) * 0.9626 AS rhs;
```

### 4-3. 검증 후 발견될 수 있는 이슈 (예상)

- 식권 보유 쿠폰 라디오 — 같은 (amount, source) 그룹에서 FIFO (가장 오래된 row 먼저 사용). 만료 임박 우선이 아니라 created_at 오름차순. 운영진이 만료 임박 우선을 원하면 `lib/coupons.ts:fetchAvailableCouponsByPhone` 의 sort 변경.
- 정산 페이지 default 기간 = 5/15~5/17. 테스트 데이터가 그 외 일자에 있으면 사용자가 직접 기간 조정해야 보임.
- 매장별 정산 — 다부스 결제의 PG거래액/쿠폰차감 분배는 **부스 subtotal 비례**. 식권 분배(`voucher_consumed`)는 v3 의 비례+잔여흡수 룰 그대로 부스 row 에 저장된 값을 합산. 정합성은 `voucher_consumed` 만 따로 계산되므로 어긋날 일 없음.

## 6. 정산 검증용 통합 SQL (참고)

```sql
-- v3+v4 통합 검증 (식권 결제 후)

-- A. 부스별 식권 분배 정합성
SELECT
  p.id AS payment_id,
  c.discount_amount AS voucher_amount,
  SUM(o.voucher_consumed) AS sum_consumed,
  SUM(o.voucher_burned) AS sum_burned,
  CASE
    WHEN c.type = 'meal_voucher'
      AND SUM(o.voucher_consumed) + SUM(o.voucher_burned) = c.discount_amount
    THEN 'OK' ELSE 'MISMATCH'
  END AS voucher_balance
FROM payments p
JOIN orders o ON o.payment_id = p.id
LEFT JOIN coupons c ON c.id = p.coupon_id
WHERE p.coupon_id IS NOT NULL AND c.type = 'meal_voucher'
GROUP BY p.id, c.type, c.discount_amount;

-- B. 정산 정합성 (매장송금 = PG입금 + 운영자 순지출)
SELECT
  ROUND(SUM(o.subtotal) * 0.9626, 2) AS booth_payout,
  ROUND(SUM(p.total_amount) * 0.9626, 2) AS organizer_pg_in,
  ROUND((SUM(o.subtotal) - SUM(p.total_amount)) * 0.9626, 2) AS organizer_loss,
  ROUND(SUM(o.subtotal) * 0.9626
       - SUM(p.total_amount) * 0.9626
       - (SUM(o.subtotal) - SUM(p.total_amount)) * 0.9626, 2) AS diff_should_be_zero
FROM orders o JOIN payments p ON p.id = o.payment_id
WHERE p.status = 'paid' AND o.status NOT IN ('cancelled');

-- C. 식권 잔여 추적 (전화번호별)
SELECT phone, discount_amount, source,
  COUNT(*) FILTER (WHERE status='active' AND expires_at > NOW()) AS remaining,
  COUNT(*) FILTER (WHERE status='used') AS used_count
FROM coupons WHERE type='meal_voucher'
GROUP BY phone, discount_amount, source
ORDER BY phone;
```
