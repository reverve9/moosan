#### C-8. fix v2 — 중복 발송 + pending 잔류 (`1cc9092`)
- 1차 통합 테스트에서 픽업 알림톡 **2개 발송** + `alimtalk_logs.status='pending'` 잔류 관찰.
- 원인:
  - `SEND_TIMEOUT_MS=3s` 너무 짧음. Solapi 서버는 발송 완료했지만 우리는 timeout 판정 → `isRetriable=true` (msg 에 "timeout") → 재시도 → 중복 발송. `withTimeout` 은 Promise.race 로 reject 만 시킬뿐 실 호출 cancel 불가 (SDK AbortSignal 미지원).
  - `res.json()` 후 await 패턴은 Vercel Node serverless 에서 함수 suspend 가능. 최종 UPDATE 못 끝내고 종료 → 'pending' 잔류. (핸드오프 doc 의 fire-and-forget 가정 틀림.)
- 수정:
  - `SEND_TIMEOUT_MS` 3s → 8s (Solapi 정상 응답시간 + 마진. Vercel maxDuration 10s 안전 여유)
  - 재시도 로직 완전 제거 — timeout = "보냈는지 모름" 이라 재시도 X. Solapi 자체 재시도/큐에 위임. `isRetriable` 함수 삭제, `attempt` 변수 인라인.
  - `response_payload` 직렬화 방어 `toSafeJson(value)` — effect-ts 스키마 메타데이터로 supabase-js JSON.stringify throw 케이스 대비. `try { JSON.parse(JSON.stringify(value)) } catch { fallback }`.
  - 3개 endpoint (`booth-orders/ready`, `orders/cancel`, `payments/cancel`) 모두 `await ... .catch(...)` 를 `res.status(200).json()` **앞으로 이동**. 응답 +수초 지연 trade-off 로 UPDATE 완료 보장.

## 6. 알려진 제약 / 운영 메모

### 6-1. 알림톡 응답 시간
- 부스앱 [준비완료] 버튼 → 응답 2-5초 소요 (Solapi await 때문). 정상 동작이고 부스 직원 체감 영향 미미.
- 어드민 환불도 동일 — pg 일 때 Toss + Solapi 둘 다 직렬이라 가장 느림 (~5-8초).

### 6-2. 멱등성 한계
- `idempotency_key UNIQUE` 가 같은 (order_id, booth_id, template_type) 조합 재발송 차단. **다른 동일 의미 발송도 차단됨** — 예: 픽업 후 다시 픽업 처리 (실제로는 ready_at 멱등이라 이미 차단되지만, 코드 흐름상).
- DELETE 로 row 정리하면 재발송 가능 — 운영중 필요 시 admin SQL 로 처리.

### 6-3. 'pending' 잔류 시 의미
- 현재 코드는 응답 전 await 이라 'pending' 거의 안 남음. 만약 남으면 Vercel 함수가 UPDATE 도중 강제 종료 (10s maxDuration 초과 등) 가능성.
- 'pending' 행은 idempotency_key UNIQUE 로 후속 발송 차단. 운영 영향 시 SQL DELETE.

### 6-4. Solapi 응답 직렬화
- `response_payload` 는 `toSafeJson()` 으로 박제. effect-ts 스키마 메타데이터 제거. raw response 분석 필요 시 솔라피 콘솔 또는 Vercel 함수 로그 참조.

### 6-5. phone.ts 사본
- `src/lib/phone.ts` (client) + `api/_lib/phone.ts` (server) 에 `normalizePhone` 동일 로직 사본 존재. **정규화 로직 변경 시 두 파일 동시 수정**. 추후 정리 시 `phone-pure.ts` + `phone-storage.ts` 분리 → 공통 path 이동 가능.

### 6-6. 솔라피 ENV 보호
- `SOLAPI_API_KEY/SECRET` 절대 client 노출 금지 — `VITE_` prefix 붙이지 말 것.
- `.gitignore` 패턴 `.env.* + !.env.example` 로 모든 환경별 .env 차단.
- 행사 종료 후 API key rotation 권장 (콘솔에서 새 키 발급 → Vercel ENV 교체 → 구 키 비활성화).

### 6-7. 검수중 템플릿 자동 skip
- `KAKAO_TEMPLATE_REFUND` env 빈 값 또는 미설정 → `status='skipped_no_template'` 자동 로깅. 검수 통과 후 env 채우면 정상 발송 활성. 코드/배포 변경 불필요.

### 6-8. 빌드 검증
- `npm run build` (= `tsc -b && vite build`) 가 prod 동등 검증. 로컬 `tsc --noEmit` incremental cache 가 가끔 noUnusedLocals 등 놓침. 푸시 전 캐시 클리어 권장:
  ```bash
  rm -f *.tsbuildinfo && npx tsc -b
  ```
