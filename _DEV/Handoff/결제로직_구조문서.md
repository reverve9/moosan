# 결제 로직 구조 문서 (Moosan)

> 푸드 부스 푸드코트형 축제 앱의 결제/주문/정산 구조.
> 토스페이먼츠 + Supabase(Postgres + Realtime) + Vercel Edge Functions.
> 다른 축제 앱 구조 변경 시 참고용 레퍼런스.

```
    ├─ /checkout/success        │                        │
    └─ /order/:id (Realtime)    │                        │
                                │                        │
       ────────── Supabase (Postgres + Realtime) ──────────
                                │
                                ├─ payments / orders / order_items
                                ├─ food_booths / food_menus

                                │
```

## 2. 핵심 DB 스키마

### 2.1 `payments` (결제 1건 단위)

```sql
CREATE TABLE payments (
  id UUID PRIMARY KEY,
  toss_order_id TEXT UNIQUE NOT NULL,   -- Toss 에 넘기는 orderId (P-00000001)
  payment_key TEXT,                     -- Toss 승인 후 저장
  phone TEXT NOT NULL,
  total_amount INTEGER NOT NULL,        -- 할인 적용 후 실결제액
  discount_amount INTEGER DEFAULT 0,
  coupon_id UUID REFERENCES coupons(id),
  refunded_amount INTEGER DEFAULT 0,    -- 누적 환불액
  status TEXT CHECK (status IN ('pending','paid','cancelled')),
  paid_at TIMESTAMPTZ,
  cancelled_at TIMESTAMPTZ,
  meta JSONB DEFAULT '{}',              -- cancel_reason, cancelled_via 등
  created_at TIMESTAMPTZ,
  updated_at TIMESTAMPTZ
);
```

**트리거**: `BEFORE INSERT` 시 `toss_order_id` 를 전역 sequence `payment_toss_order_seq` 에서 `P-00000001` 형식으로 자동 채움 (DELETE 무관).

### 2.2 `orders` (부스 scope 단위 — 결제 1건 당 N건)

```sql
CREATE TABLE orders (
  id UUID PRIMARY KEY,
  payment_id UUID REFERENCES payments(id) ON DELETE CASCADE,
  order_number TEXT UNIQUE NOT NULL,    -- A01-0428-0001 (booth_no-MMDD-NNNN)
  booth_id UUID REFERENCES food_booths(id),
  booth_no TEXT NOT NULL,
  booth_name TEXT NOT NULL,
  subtotal INTEGER NOT NULL,            -- 이 부스 라인 합계 (쿠폰 할인 전)
  phone TEXT NOT NULL,
  status TEXT CHECK (status IN ('pending','paid','confirmed','completed','cancelled')),
  paid_at TIMESTAMPTZ,                  -- 결제 승인 시각 (elapsed 기준)
  confirmed_at TIMESTAMPTZ,             -- 부스 "확인" 시각
  ready_at TIMESTAMPTZ,                 -- 부스 "준비완료" 시각
  estimated_minutes INTEGER,            -- 조리 예상 분
  cancelled_at TIMESTAMPTZ,
  cancel_reason TEXT,
  cancelled_by TEXT CHECK (cancelled_by IN ('booth','admin')),
  meta JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ
);
```

**트리거**: `BEFORE INSERT` 시 `booth_order_counters` 에서 부스별 누적 카운터를 `advisory_xact_lock` 으로 직렬화해서 `order_number` 자동 채움.

### 2.3 `order_items` (순수 메뉴 라인)

```sql
CREATE TABLE order_items (
  id UUID PRIMARY KEY,
  order_id UUID REFERENCES orders(id) ON DELETE CASCADE,
  menu_id UUID REFERENCES food_menus(id),
  menu_name TEXT NOT NULL,       -- 스냅샷 (메뉴 삭제돼도 유지)
  menu_price INTEGER NOT NULL,
  quantity INTEGER CHECK (quantity > 0),
  subtotal INTEGER NOT NULL,
  created_at TIMESTAMPTZ
);
```

### 2.4 `booth_order_counters` (부스별 누적 카운터)

```sql
CREATE TABLE booth_order_counters (
  booth_id UUID PRIMARY KEY,
  last_no INTEGER DEFAULT 0,
  updated_at TIMESTAMPTZ
);
```

DELETE 영향 받지 않음. 같은 부스 동시 INSERT 는 `pg_advisory_xact_lock(hashtext('booth_order_' || booth_id))` 로 직렬화.

### 2.5 `coupons` (수동/설문 발급)

```sql
CREATE TABLE coupons (
  id UUID PRIMARY KEY,
  code TEXT UNIQUE NOT NULL,            -- MS-XXXXXX (6자리, 0/1/O/I 제외)
  discount_amount INTEGER CHECK (> 0),  -- 고정 금액 할인
  min_order_amount INTEGER DEFAULT 10000,
  status TEXT CHECK (status IN ('active','used')),
  issued_source TEXT CHECK (IN ('manual','survey')),
  issued_phone TEXT,                    -- 수동 발급 대상 번호 (선택)
  phone TEXT,                           -- 설문 자동발급 번호 (자동 적용 키)
  expires_at TIMESTAMPTZ,
  used_at TIMESTAMPTZ,
  used_payment_id UUID REFERENCES payments(id),
  festival_id UUID,
  note TEXT,
  created_at TIMESTAMPTZ
);
```

### 2.6 `food_booths` · `food_menus` · `booth_accounts`

```sql
-- 부스 상태 토글
food_booths.is_open    BOOLEAN    -- false = 영업종료
food_booths.is_paused  BOOLEAN    -- true  = 일시 준비중
food_menus.is_sold_out BOOLEAN    -- 품절 토글

-- 부스 로그인 (bcrypt)
booth_accounts(id, booth_id, login_id, password_hash)
```

### 2.7 뷰: `booth_waiting_counts`

```sql
CREATE VIEW booth_waiting_counts AS
SELECT fb.id booth_id, COUNT(o.id) waiting_count
FROM food_booths fb
LEFT JOIN orders o
  ON o.booth_id = fb.id
 AND o.status = 'paid'
 AND o.confirmed_at IS NULL
 AND o.created_at > now() - INTERVAL '3 hours'
WHERE fb.is_active = true
GROUP BY fb.id;
```

### 2.8 Realtime publication

```sql
ALTER PUBLICATION supabase_realtime ADD TABLE
  payments, orders, order_items, coupons, food_booths, food_menus;

ALTER TABLE payments      REPLICA IDENTITY FULL;
ALTER TABLE orders        REPLICA IDENTITY FULL;
ALTER TABLE order_items   REPLICA IDENTITY FULL;
ALTER TABLE food_booths   REPLICA IDENTITY FULL;
```

**중요**: non-PK 컬럼 필터(`booth_id=eq.X`, `payment_id=eq.Y`)가 UPDATE 이벤트에서도 동작하려면 `REPLICA IDENTITY FULL` 필수. 기본 DEFAULT 는 PK 만 포함.

### 4.3 토스 SDK 초기화 `src/lib/toss.ts`

```ts
import { loadTossPayments } from '@tosspayments/payment-sdk'

const CLIENT_KEY = import.meta.env.VITE_TOSS_CLIENT_KEY
let tossPaymentsPromise = null

export function getTossPayments() {
  if (!tossPaymentsPromise) {
    tossPaymentsPromise = loadTossPayments(CLIENT_KEY)  // lazy, 최초 1회
  }
  return tossPaymentsPromise
}

export async function confirmPayment({ paymentKey, orderId, amount }) {
  const res = await fetch('/api/payments/confirm', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ paymentKey, orderId, amount }),
  })
  const data = await res.json()
  if (!res.ok) throw new Error(data?.message || '결제 승인 실패')
  return data
}
```

## 5. 서버 API (Vercel Functions)

모두 `api/*.ts` → `@vercel/node` handler. Toss Basic Auth 는 `base64(secretKey + ':')`.

### 5.1 `POST /api/payments/confirm` — 결제 승인

**파일**: `api/payments/confirm.ts`

```ts
Body: { paymentKey, orderId, amount }

→ Toss POST https://api.tosspayments.com/v1/payments/confirm
  headers: { Authorization: 'Basic <base64(secret:)>' }
  body:    { paymentKey, orderId, amount }

→ 응답 그대로 전달 (200 / 4xx)
```

DB 업데이트는 **하지 않음**. 승인 성공 후 클라이언트가 `markPaymentPaid` 호출.

### 5.2 `POST /api/payments/cancel` — 어드민 풀 환불

**파일**: `api/payments/cancel.ts`

```ts
Body: { paymentId, reason }

1) payments + orders 조회
2) 적격성 검증
   - payment.status = 'paid'
   - 잔액 (total_amount - refunded_amount) > 0
   - 이미 cancelled 인 부스 order 는 무시 (이력 보존)
   - 남은 paid orders 중 confirmed_at 또는 ready_at 있으면 거부
     (ORDER_ALREADY_CONFIRMED)
3) cancelAmount = 잔액
   Toss POST /v1/payments/{payment_key}/cancel
4) DB 업데이트
   - payments: status='cancelled', refunded_amount=total_amount,
               meta.cancel_reason, meta.cancelled_via='admin'
   - 남은 paid orders: status='cancelled', cancelled_by='admin'
     (이미 cancelled 인 부스 order 는 건드리지 않음)
```

### 5.3 `POST /api/orders/cancel` — 부스 거절 · 부분 환불

**파일**: `api/orders/cancel.ts`

```ts
Body: { orderId, reason }

1) order + parent payment 조회
2) 적격성 검증
   - order.ready_at IS NULL  (조리완료되면 거절 불가)
   - order.status in ('paid','confirmed')
   - payment.status = 'paid' AND payment.payment_key NOT NULL
3) refundAmount = min(order.subtotal, payment.total_amount - refunded_amount)
   · 쿠폰 할인은 운영자 부담 → 영업점은 subtotal 그대로 환불 (비율 분배 X)
   · 큰 쿠폰 edge case cap
4) Toss POST /v1/payments/{payment_key}/cancel (cancelAmount=refundAmount)
5) DB 업데이트
   - orders[id]: status='cancelled', cancelled_by='booth', cancel_reason
   - payments: refunded_amount += refundAmount
               누적 >= total_amount 면 payments.status='cancelled'
```

### 5.4 `POST /api/coupons/validate` — 쿠폰 검증 (read-only)

**파일**: `api/coupons/validate.ts`

```ts
Body: { code, subtotal }

검증:
  - 쿠폰 존재
  - status = 'active'
  - expires_at > now()
  - subtotal >= min_order_amount

응답: { valid: true, couponId, code, discount, finalAmount }

※ 이 API 는 상태 변경 안 함. 실제 used 전이는 markPaymentPaid 에서.
```

## 6. 주문/결제 상태 전이

### 6.1 `payments.status`

```
pending ──[/api/payments/confirm 성공 + markPaymentPaid]──► paid
                                                             │
                                                             │ [/api/payments/cancel 또는
                                                             │  모든 orders 부스거절로 누적환불=total]
                                                             ▼
                                                         cancelled
```

### 6.2 `orders.status`

```
pending ──[markPaymentPaid]──► paid ──[confirmBoothOrder]──► confirmed
                                 │                              │
                                 │        [markBoothOrderReady] ▼
                                 │                          completed
                                 │
                                 └─[/api/orders/cancel]──► cancelled
                                                          (paid 또는 confirmed 상태 + ready_at NULL)
```

### 6.3 시간 컬럼 기준

| 컬럼 | 채워지는 시점 | 용도 |
|------|-------------|------|
| `orders.created_at` | INSERT (결제하기 클릭) | 정렬, DB 이력 |
| `orders.paid_at` | `markPaymentPaid` | **부스/어드민 elapsed 기준** |
| `orders.confirmed_at` | 부스 "확인" | 조리중 판정 |
| `orders.ready_at` | 부스 "준비완료" | 픽업 알림 |
| `orders.cancelled_at` | 거절 처리 | 거절 이력 |

> `paid_at` 이 필요한 이유: `created_at` 기준이면 Toss 인증/승인 30~40초가 포함돼 elapsed 알람이 조기 발화됨. `18_orders_paid_at.sql` 참고.

```ts
채널 2: order_items (booth 필터 불가 · 전체 구독)
  - 모든 변경 → onChange() refetch
```

## 11. 주요 설계 결정 · 주의사항

1. **1 Toss 결제 = 1 payments + N orders (부스별)**
   푸드코트형 구조. 손님은 한 번 결제하지만 부스마다 독립 주문. 부스별 거절/환불/조리관리가 독립적으로 가능.

2. **트리거로 `order_number`, `toss_order_id` 자동 생성**
   클라이언트가 숫자 생성 책임지지 않음. `advisory_xact_lock` + 카운터 테이블로 race-safe. DELETE 해도 번호 감소 없음.

3. **서버에서 Toss 호출 전담**
   `TOSS_SECRET_KEY` 는 Vercel Function 에서만 접근. 클라는 SDK 의 `loadTossPayments` + `requestPayment` 만.

4. **금액 위변조 방지**
   success 페이지에서 `payment.total_amount === query.amount` 검증 후에만 confirm 호출.

5. **쿠폰 할인 분배 정책**
   부스 거절 환불액 = `order.subtotal` (쿠폰 할인 비율 배분 X). 쿠폰 할인은 운영자(주최) 부담이라는 정책.

6. **Realtime 필터 for non-PK**
   `REPLICA IDENTITY FULL` 없으면 UPDATE 이벤트에서 `booth_id=eq.X` 같은 필터가 누락됨.

7. **paid_at 분리**
   `created_at` 은 결제시도 시점, `paid_at` 은 승인완료 시점. elapsed 알람/정렬은 `paid_at` 기준.

8. **StrictMode 중복 호출 방지**
   success 페이지는 `useRef` 로 1회 실행 보장 (Toss confirm 중복 호출 시 `ALREADY_PROCESSED_PAYMENT` 에러).

9. **history.pushState 로 뒤로가기 차단**
   결제 직후 `/order/:id?from=checkout` 에서 popstate 리스너로 메뉴 페이지로 리다이렉트 (Toss 도메인으로 돌아가는 사고 방지).

10. **windowTarget: 'self'**
    모바일에서 Toss 결제창이 새 탭/팝업 없이 현재창에서 리다이렉트되도록 필수. 데스크탑은 iframe 폴백.
