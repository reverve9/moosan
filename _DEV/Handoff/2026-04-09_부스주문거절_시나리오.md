# 부스 주문 거절 + 부분 환불 — 운영 테스트 시나리오

배포 후 검증용. 마이그 17 (`_DEV/Seeds/17_order_rejection.sql`) 적용 완료가 전제.

## 사전 점검

1. Supabase Studio SQL editor 에서 마이그 17 적용 확인

   SELECT column_name FROM information_schema.columns

2. 손님 / 부스 / 어드민 세 도메인 모두 PWA 재방문 (서비스워커 캐시 갱신)

## 시나리오 1 — 부분 환불 (멀티 부스)

**목적**: 한 결제 안에 두 부스가 있을 때 한 부스만 거절 → 다른 부스 정상 진행 + 부분 환불 검증.

1. **손님**: `/program/food` 진입, 부스 A 메뉴 1개 (예 5,000원) + 부스 B 메뉴 1개 (예 8,000원) 카트에 담기
2. **손님**: `/checkout` → 휴대폰 입력 → 결제 → Toss 결제 → `/order/:id?from=checkout` 도달
3. **부스 A 대시보드**: 새 카드 등장 (부스 A subtotal 5,000원)
4. **부스 A**: 카드의 빨간 **거절** 버튼 클릭
5. 사유 모달: "재료 소진" 선택 → **거절 + 환불** 클릭
6. 토스트 "[A01-MMDD-NNNN] 거절 + 환불 처리됐어요" 노출, 카드 사라짐

### 손님 화면 (`/order/:id`) 즉시 반영
- 상단 상태 카드: "일부 취소 — 일부 매장이 주문을 거절해 환불 처리됐어요"
- 결제 정보 블록에 **환불금액 -5,000원** 줄 추가 노출
- 부스 A 카드:
  - 빨간 보더 + 핑크 배경
  - 상단 헤더의 상태 배지가 "취소됨"
  - "거절 사유 / 재료 소진 / 5,000원 환불 처리됐어요" 박스
  - 메뉴 아이템 행은 회색 + 취소선
- 부스 B 카드: 그대로 정상 (waiting → preparing → ready 흐름 진행)

### 어드민 대시보드 즉시 반영
- **모니터** (`/monitor`): 부스 A 카드의 미확인 카운트에서 자동 제거 (이미 부스 거절했으니)
- **주문/결제 관리** (`/orders`):
  - 해당 결제 행에 "결제완료 + 부분취소" 배지 노출
  - 상단 totals: 환불 박스에 +5,000원 누적
  - 매출(원) 박스에서 5,000원 차감 반영
  - 행 클릭 → 상세 모달:
    - 결제정보에 "환불 금액 -5,000원" + "잔액 7,000원"
    - 부스 A 카드 상단에 "[부스 거절] 재료 소진" 빨간 라인
    - "환불 처리" 섹션의 버튼이 "잔액 7,000원 환불" 로 표시 (남은 부스 B 가 paid AND confirmed_at=null 이면)
- **대시보드** (`/`) → 매출관리 탭: 매출 KPI / 시간대 / 객단가 모두 차감 반영

## 시나리오 2 — 부스 확인 후 거절

**목적**: confirmed 상태 (부스가 "확인" 누른 후) 에서도 거절 가능 검증.

1. 손님이 부스 A 단독 결제 (5,000원)
2. 부스 A: "확인" 클릭 → status='confirmed', 카드는 inProgress 색
3. 부스 A: 거절 버튼 → 사유 "조리 불가" → 확인
4. Toss 환불 → orders.status='cancelled' + payments.status='cancelled' (단독 부스라 누적 환불액 = total)

### 손님 화면
- 상태 카드: **"취소됨 — 주문이 취소되었습니다"** (partial 이 아니라 cancelled)
- 부스 A 카드: 빨간 + 거절 사유 박스

### 어드민
- 결제 행 자체가 "취소됨" 상태 + 회색 행
- totals: cancelled 카운트 +1, 매출 0

---

## 시나리오 3 — 거절 불가 (조리 완료 후)

**목적**: ready_at 채워진 주문은 거절 불가.

1. 부스 A: 새 주문 → "확인" → "조리완료" 클릭 (status='completed', ready_at NOT NULL)
2. 카드는 우측 완료 리스트로 이동 — 거절 버튼 자체가 노출 안 됨
3. (만약 API 직접 호출로 우회 시도해도) 서버가 `ORDER_ALREADY_READY` 로 409 거부

---

## 시나리오 4 — 부스 거절 후 어드민 풀환불

**목적**: 한 부스만 거절된 부분환불 상태에서 어드민이 잔액 풀환불 가능 여부.

1. 시나리오 1 직후 (부스 A 거절, 부스 B 는 아직 paid + confirmed_at=null)
2. 어드민 → 주문/결제 관리 → 해당 결제 행 클릭
3. 환불 처리 섹션에서 "잔액 7,000원 환불" 클릭
4. 사유 입력 → 확인
5. Toss 잔액 환불 → 부스 B 도 cancelled (cancelled_by='admin') + payments.status='cancelled'

### 손님 화면
- 상태 카드: **"취소됨"** (partial → cancelled 전이)
- 부스 A: "[부스 거절] 재료 소진"
- 부스 B: "[어드민 환불] (어드민 입력 사유)"
- 환불금액 -12,000 (= 5,000 + 7,000)

### 어드민 거부 시나리오
- 부스 B 가 이미 "확인" 눌렀으면 → "이미 매장에서 확인한 주문이 포함되어 있어 취소할 수 없습니다" 거부
- 부스 단위로 거절은 부스 B 본인이 직접 해야 함

---

## 시나리오 5 — 큰 쿠폰 edge case (선택)

**목적**: 결제 잔액 < 부스 subtotal 인 비현실적 케이스에서 cap 동작 검증.

1. 5,000 + 8,000 = 13,000 카트
2. 쿠폰 -10,000 적용 → 결제 3,000
3. 부스 A 거절 → 환불액 = `min(5,000, 3,000)` = **3,000**
4. payments.refunded_amount=3,000, status='cancelled' (잔액 0 도달)
5. 부스 B 자동으로 cancelled 안 됨 (B 는 자기가 진행하거나 별도 거절해야 함)
6. payments 가 cancelled 가 됐으므로 어드민 잔액 환불은 더 이상 불가
7. 운영자 입장에서 부스 B 는 페스티벌 외 정산으로 처리 필요 (실제 운영에서 거의 안 발생)

> **참고**: 운영 중엔 쿠폰이 보통 1,000~2,000원이라 이 케이스는 거의 발생 안 함. 발생 시 운영자가 별도 처리.

## 회귀 점검 (기존 흐름 깨지지 않는지)

- [ ] 손님이 정상적으로 단일 부스 결제 → 부스 확인 → 조리완료 흐름 (거절 없이) 정상
- [ ] 어드민이 결제 직후 (부스 미확인) 풀환불 → 정상
- [ ] AdminMonitor 카운트가 부스 거절 후 자동 감소
- [ ] AdminDashboard 매출관리 KPI / 시간대 / 부스/메뉴 차트 모두 부분환불 차감 반영
- [ ] 부스 PWA 헤더의 connection dot / banner — 거절 액션 중에도 정상
- [ ] OrderStatusPage 새로고침 후에도 거절 사유 / 환불금액 그대로 (DB 에 저장됨)
