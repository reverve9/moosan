### 3-2. 정산 공식 (확정)

```typescript
function calcVoucherSettlement(orderTotal: number, voucherAmount: number) {
  const consumed = Math.min(voucherAmount, orderTotal)
  const burned = voucherAmount - consumed
  const userPaid = Math.max(0, orderTotal - voucherAmount)
  return {
    voucherConsumed: consumed,    // orders.voucher_consumed
    voucherBurned: burned,        // orders.voucher_burned
    userPaid: userPaid,           // PG로 결제할 금액
    vendorSettlement: orderTotal, // 매장 정산 = consumed + userPaid
  }
}
```

### 3-3. 미해결 설계 결정 (v3 진입 시 짚을 것)

1. **부스가 여러 개일 때 식권 적용 정책** — 한 결제 = 부스 N개 (`orders` row N개) 인데 핸드오프는 `orders.voucher_consumed` 단일 컬럼. 부스별로 분배할지, 첫 부스에 몰아넣을지, 아니면 식권은 단일 부스 결제만 허용할지 결정 필요.
2. **전액 식권 결제 (`userPaid=0`) 의 `payments` row** — 토스 호출 스킵하면 `payment_key`/`toss_order_id` 가 없거나 placeholder. 트리거가 `toss_order_id` 자동 채우므로 충돌 없음. `payment_key` 는 NULL 유지. status 는 어떻게 둘지 (`paid` 직행? 별도 path?). markPaymentPaid 가 paymentKey 인자 받으므로 nullable 처리 필요.
3. **환불 시 식권 복구** — 핸드오프에서는 본 페이즈 외, 운영진 수동. 그러나 부스 단위 환불 (세션 27) 이 들어가있어서 구체적인 거동 정의 필요할 수 있음.
4. **식권 자동 추천 vs 사용자 선택** — 핸드오프 §6 "사용자 선택 (자동 추천 X)". 다만 보유 쿠폰이 1개면 default 선택 둘지.

### 3-4. 손대기 전 추천 작업 순서

4. `handleSubmit` — `userPaid === 0` 분기: PG skip 후 직접 `markPaymentPaid` 류 호출 → `/order/:paymentId` 이동.
