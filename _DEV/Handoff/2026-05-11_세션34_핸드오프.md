### 6-1. 환불 비례 분배 — 끝수 보정 위치
- 멀티부스 쿠폰 결제에서 각 부스 환불액은 `floor(subtotal × total_amount / sum_subtotal)`
- 마지막 살아있는 부스 환불 시 `remaining` 전체 → 끝수 흡수 → 누적 환불액 = total_amount 보장
- `payment.status='cancelled'` 판정은 **부스 카운트 기반**(`isLastLiveBooth`). amount 비교 X — 비례 floor 로 누적 환불액 < total_amount 인 채 마지막 부스 진입 가능

### 6-2. 식권 멀티부스 부분 취소 — 손님 보상은 운영진 수동
- 운영 정책: 식권 부분 취소 = 운영진 수동 처리 (식권 재발급 등 케이스별)
- 시스템은 정산에 `voucherBurned` 로 누적해 운영자 손실 가시화만

### 6-3. 식권 순차 그리디 — 부스 정렬 영향
- `calcVoucherSettlement` 가 booths 인자 순서대로 그리디 → 호출자가 정렬 책임
- CheckoutPage / HelpDeskOrderTab 의 `groups` 배열 순서 = 카트 등장 순서. 향후 우선순위 정렬 필요시 호출 측 변경

### 6-4. 0원 결제 방어 로직
- `total_amount === 0` 케이스: cancel.ts 양쪽에서 payment_key 검증 skip + Toss 호출 skip + 환불 알림톡 skip
- voucher_only 분기와 별도로 작동 → 과거 `'pg'` 로 저장된 0원 결제도 정상 처리

### 6-5. 정산 cancelled 식권 처리 정합성
- `fetchSettlementRawData` 가 `payments.status='paid'` 필터 → 전체 취소된 결제는 통계 제외
- 부분 취소된 결제의 cancelled 부스만 `aggregateByDay` 가 voucherBurned 로 회수
- 매장별 탭은 cancelled 제외 → 일별 탭 totals 와 약간 불일치 가능 (다음 세션 KPI 추가 고려)

### 6-6. 정산 기본 날짜 정책
- 기본 dateFrom/dateTo = `todayKstString()` — 운영중에도 dev 테스트에도 합리적
- 행사 기간 종합 조회는 사용자가 직접 dateFrom/dateTo 조정

### 6-7. 천단위 구분기호 적용 범위
- `exportToExcel` / `exportToExcelMultiSheet` 양쪽에 적용 → 모든 어드민 엑셀에 일괄 적용 (정산뿐 아니라 주문/결제, 쿠폰 등)
- 셀 타입 `'n'` (number) 만 포맷 적용 → 텍스트는 영향 없음
