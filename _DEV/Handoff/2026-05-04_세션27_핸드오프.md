## 1. 세션 27 에서 끝난 것

### A. 주문/결제 관리 — 부스 단위 행 분리 + 부스별 환불

**문제** — 한 결제에 여러 부스가 묶이면 어드민 리스트가 결제 1줄로만 떠서 매장 정산이 어려움.

**변경 요지** — 결제 단위 행 → 부스 단위 행으로 펼침. 어드민 환불도 결제 풀환불 UI 폐기하고 부스별/일괄 환불로 교체.

**서버 — `api/orders/cancel.ts`**
- `cancelledBy?: 'booth' | 'admin'` 파라미터 신설 (미지정 시 'booth' — 부스 클라이언트 호환)
- DB `orders.cancelled_by` 에 그대로 저장 → 모달 라벨 "부스 거절" / "어드민 환불" 자동 분기

**클라이언트 — `src/lib/adminPayments.ts`**
- 결제 단위 풀환불 함수 제거: `cancelPayment` / `isRefundable` / `remainingRefundable` 삭제
- 부스 단위 환불로 교체:
  - `BoothOrderRow` 타입 — `payment` + 단일 `order` + `siblingCount` (같은 결제 묶음 수)
  - `fetchPaymentsList` 가 결제 → 부스 단위 행으로 펼쳐서 반환 (결제 최신순 → booth_no 오름차순)
  - `isBoothOrderRefundable(detail, orderId)` — 부스 거절과 동일 조건 (paid/confirmed + ready_at IS NULL + 잔액>0)
  - `boothOrderRefundAmount(detail, orderId)` — `min(order.subtotal, remaining)`
  - `refundBoothOrder(orderId, reason)` — `/api/orders/cancel` 을 `cancelledBy='admin'` 호출

**UI — `src/pages/admin/AdminOrders.tsx`**
- 테이블 행: 부스 단위 1줄. 같은 결제로 여러 부스면 "동시결제 N건" 보조 라벨 노출
- 금액 컬럼: `payment.total_amount` → `order.subtotal` (해당 매장 정산 기준 금액)
- 상태: 결제 취소면 "취소됨", 부스만 거절이면 "매장거절" 별도 표기
- 통계: 매출은 살아있는 부스 주문 subtotal 합. 쿠폰 할인/환불 금액은 결제 ID 기준 중복 제거. 결제/취소 카운트는 부스 행 기준
- 엑셀 export 도 부스 단위 행

**DetailModal**
- 부스 카드마다 우측 하단 빨간 "환불" 버튼 + 환불 가능 금액 표시
- 환불 불가 시 "이미 환불됨" / "조리완료 - 환불 불가" / "확인됨 - 환불 불가" 회색 텍스트
- 환불 사유 입력은 모달 상단 단일 textarea (모든 환불 액션 공유)
- **일괄 환불 버튼**: 환불 가능한 부스가 2개 이상일 때만 노출. 직렬 처리(Toss/DB race 방지). 일부 실패 시 detail 재로드 후 에러 표시. 호박색(`#B45309`) 으로 개별 환불(빨강)과 시각 구분

## 8. 설계 결정 메모

### 부스별 환불을 결제 풀환불 대체로 채택한 이유
풀환불 UI 와 부스별 환불 UI 를 둘 다 두면 모달이 복잡해지고 "둘이 뭐가 다른지" 혼란. 풀환불은 결국 "환불 가능한 모든 부스를 환불"의 합과 동치 → **일괄 환불** 버튼 하나로 대체. UI 모델은 부스 단위 1축으로 통일됨.

### 일괄 환불을 직렬 처리로 둔 이유
같은 `payment_key` 에 대한 동시 Toss cancel 호출은 보장되지 않음 + DB `refunded_amount` 누적이 race 가 될 수 있어 안전하게 한 건씩. 환불 가능 부스가 보통 2~3개라 직렬도 충분히 빠름. 일부 성공 후 실패 시 `reload()` 로 최신 상태 복구.

### 카테고리 테이블/컬럼명을 그대로 둔 이유
`food_categories` → `food_zones` rename + `food_booths.category` → `zone` 컬럼명 변경은 코드 영향이 광범위 (lib, types, 기존 시드). 의미는 어긋나지만 라벨만 "구역"으로 갈아서 사용자 노출은 깔끔하고, 변수명은 운영 후 시간 여유 생기면 rename. **rename 안 함의 비용 = 신규 개발자 혼란 1회**, **rename 비용 = 코드 영향 광범위 + 마이그레이션 + QA**. 트레이드오프 의식적 선택.

### 메뉴 태그를 마스터 없이 자유 입력으로 둔 이유
- 사용자가 "카테고리가 아니니까" 명시 — 분류 의도 아님
- 카테고리 마스터 + 어드민 관리 UI = 작업량 증가, 운영자가 "태그 마스터 정의 → 메뉴에 부착"의 2단계 흐름 필요
- 자유 입력 + 자동완성으로 표기 통일 80% 해결 (전체 매장 풀에서 부분일치 추천)
- 운영 중 진짜 마스터화 필요해지면 `tags` 배열에서 unique 추출 → 마스터 테이블 시드는 1회 작업으로 가능 (역방향 호환)

### 태그 뱃지 단색 결정
- 단색 회색 = 메뉴 정보 가리지 않는 보조 표시
- 색을 태그별로 부여하면 화려해지고 시각 우선순위가 메뉴 자체와 경합
- 운영 중 "특정 태그(예: 매운맛)는 색 강조 필요" 의견 나오면 그때 화이트리스트 방식으로 추가
